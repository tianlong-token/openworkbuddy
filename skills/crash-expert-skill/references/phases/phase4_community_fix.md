# 阶段四：查找社区 fix commit 及 bug issue（详细步骤）

**⚡ 前置条件：仅当阶段三根因分类判定为"内核缺陷"或"存疑"时才执行此阶段。**

## 总览

阶段四包含三个子步骤，按顺序执行：

| 步骤 | 名称 | 前置条件 | 产出 |
|------|------|---------|------|
| 4.1 | 查找上游社区 fix commit | 始终执行 | 上游修复 commit 列表 |
| 4.2 | 搜索社区 bug/issue 报告 | 始终执行 | 相关 bug issue 列表 |
| 4.3 | 🔴 **检查发行版仓库是否已修复** | `DISTRO_GIT_REPO` 非空 | 发行版修复状态 + 修复所在 tag |

---

## 步骤 4.1：查找上游社区 fix commit

结合当前版本内核代码及内核社区上游 git 仓库，**尽最大努力**查找匹配的 fix 或优化 commit。

若用户提供了上游社区内核代码本地仓库（上游代码默认路径：`/usr/local/src/kernel/linux-stable`），可使用 `git log`、`git log --all --grep` 等命令搜索；若无本地仓库，降级为 `web_search` + `search_knowledge_base` 在线搜索。

### 🔴 步骤 4.1.1：根因驱动的搜索策略（必须遵守）

**核心原则：搜索方向由阶段三分析出的根因和触发机制决定，而不是无脑遍历调用栈的每一层。**

#### A. 搜索前的思考（必须先完成，再动手搜索）

在执行任何 `git log` / `web_search` 之前，必须先回答以下三个问题，将答案作为搜索计划写下来：

| # | 问题 | 示例 |
|---|------|------|
| 1 | **根因是什么？** | 阶段三结论|
| 2 | **哪段代码有 bug？** | 根据根因和源码分析，定位到具体的有 bug 的函数/逻辑/代码段|
| 3 | **修复应该改动什么？** | 基于根因推理预期修复方向|

**🔴 只有明确回答了上述三个问题后，才开始搜索。搜索范围聚焦于"哪段代码有 bug"和"修复应该改动什么"所指向的文件和函数。**

#### B. 搜索策略：聚焦根因，逐步扩展

**第一轮：精准搜索（围绕根因涉及的代码）**

| 搜索方法 | 说明 |
|---------|------|
| `git log --all --oneline -- <根因涉及的文件>` | 搜索根因定位到的**具体文件**（通常 1-3 个，不是调用栈的所有文件） |
| `git log --all --oneline --grep="<有bug的函数名>"` | 搜索 commit message 中提到该函数的修复 |
| `git log --all -S "<有bug的关键代码片段>" -- <对应文件>` | 搜索修改了该代码段的 commit |

**第二轮：扩展搜索（仅当第一轮未找到时）**

| 搜索方法 | 说明 |
|---------|------|
| `git log --all --oneline --grep="<根因机制关键词>"` | 搜索描述根因机制的关键词，如 `int3_emulate`、`text_poke race` |
| `web_search` 搜索 `<有bug的函数名> + "fix"/"bug"` | 在线搜索社区讨论 |
| `search_knowledge_base` 搜索错误机制关键词 | 搜索知识库中的已知问题 |

**第三轮：广泛搜索（仅当前两轮都未找到时）**

| 搜索方法 | 说明 |
|---------|------|
| 扩展到调用链中其他层级的文件和函数 | 此时才考虑更宽泛的搜索 |
| `web_search` 搜索 panic 调用栈特征 + 内核版本 | 在社区中搜索类似崩溃报告 |

**🔴 关键约束：**
- **不要在第一轮就做广泛搜索**——先用根因分析的结论精准搜索
- **时间范围**：首次搜索**不要**设置 `--after`/`--before` 时间过滤。只有在结果过多（>100 条）时才用时间范围缩小
- **修复可能不在 panic 入口子系统中**：如果阶段三的根因指向了底层基础设施，搜索应该直接聚焦底层

### 🔴 步骤 4.1.2：修复 commit 验证标准（防止误判）

**找到候选 commit 后，必须通过以下验证才能判定为"修复 commit"：**

| # | 验证项 | 通过标准 |
|---|--------|---------|
| 1 | **根因关联性** | 该 commit 修改的代码是否与阶段三分析出的根因**在因果关系上关联**？注意：修复代码**不一定在崩溃调用栈上**——竞态修复可能改的是另一条竞争路径，初始化修复可能改的是 setup 代码，锁修复可能改的是并发的写入端。关键是判断修改是否能消除导致 crash 的因果链条中的某一环 |
| 2 | **机制匹配** | 该 commit 的修复逻辑是否能从原理上消除已分析出的根因（竞态、空指针、逻辑错误等）？ |
| 3 | **代码验证** | 用 `git show <commit>` 查看具体代码变更，确认修复内容与问题对应 |
| 4 | **区分"相关"与"修复"** | "相关"commit（如同子系统的其他改进）≠"修复"commit。修复必须直接消除导致 crash 的根因 |

**🚫 常见错误（必须避免）：**
- 🚫 **未经根因分析就开始搜索**——没有先回答"哪段代码有 bug""修复应该改动什么"，导致搜索方向偏离
- 🚫 **搜索方向与根因脱节**——阶段三已指向底层机制，搜索时却只在表面子系统中搜索
- 🚫 **找到"相关"commit 就断言是"修复"**——未通过代码验证确认 commit 的修改确实能消除根因
- 🚫 **搜索时设置过窄的时间范围**——排除了早期编写但后来才 backport 的修复
- 🚫 **无脑遍历调用栈每一层**——导致搜索噪音大、结果杂，反而淹没了真正的修复

## 步骤 4.2：搜索社区 bug/issue 报告

除了搜索 fix commit，还需搜索社区中是否有与当前问题匹配的 **bug 报告或 issue 讨论**，以获取更完整的上下文信息。

**搜索范围（按优先级）：**

| 来源 | 搜索方式 | URL 模式 |
|------|---------|---------|
| **kernel.org Bugzilla** | 按关键函数名、错误信息搜索 | `https://bugzilla.kernel.org/` |
| **LKML（Linux Kernel Mailing List）** | 按关键函数名、panic 信息搜索邮件讨论 | `https://lore.kernel.org/` |
| **Red Hat Bugzilla** | 按错误信息、调用栈关键函数搜索 | `https://bugzilla.redhat.com/` |
| **Launchpad（Ubuntu）** | 按内核模块、错误信息搜索 | `https://bugs.launchpad.net/ubuntu/+source/linux` |

**搜索关键词构建策略：**
- 使用 panic 调用栈中的**关键函数名**（如 `out_of_memory`、`oom_kill_process`）
- 使用**错误信息关键字**（如 "unable to handle kernel NULL pointer dereference at"）
- 使用涉及的**内核子系统/模块名**（如 `ext4`、`mm`、`net`）
- 结合内核版本范围缩小搜索范围

**搜索工具：**
- 优先使用 `web_search` 工具进行搜索
- 也可以使用 MCP 工具 `search_knowledge_base` 搜索本地知识库

**产出要求：**
- 列出找到的相关 bug issue（标题、链接、状态、关联性说明）
- 如果 bug issue 中包含修复方案或 workaround，一并记录
- 即使未找到完全匹配的 issue，也应记录搜索过程和关键词，以便报告中说明

## 🔴 步骤 4.3：检查发行版 git 仓库是否已在新版本中修复

**前置条件：阶段一中 `DISTRO_GIT_REPO` 变量非空（即用户提供了发行版内核 git 仓库路径）。如果为空，跳过此步骤。**

当用户提供了发行版内核 git 仓库时，必须检查该仓库的**较新版本/tag 中是否已包含修复**。这对用户的决策至关重要：
- 如果发行版已修复 → 用户只需升级到对应版本即可，无需自行反向移植
- 如果发行版未修复 → 用户需要自行反向移植上游 commit 或向发行版维护者提交请求

### 🔴 核心原则：证据驱动，必须通过代码验证

**禁止仅凭 commit 标题或 git log 关键词匹配就断言"已修复"。必须验证修复代码确实存在于新版本中。**

### 步骤 4.3.1：确认当前 vmcore 对应的发行版版本

从阶段一的 `sys` 输出中提取内核版本字符串（如 `5.4.119-19.0009.54`），确定当前 vmcore 运行的发行版内核版本。

```bash
cd $DISTRO_GIT_REPO
# 查找与当前内核版本匹配的 tag
git tag | grep "<版本关键字>" | sort -V
```

### 步骤 4.3.2：在发行版仓库中搜索修复 commit

根据步骤 4.1 找到的上游修复 commit，在发行版仓库中搜索对应的 cherry-pick 或反向移植：

**方法一：按上游 commit 标题搜索**
```bash
cd $DISTRO_GIT_REPO
# 使用上游 fix commit 的标题关键词搜索
git log --all --oneline --grep="<上游 commit 标题关键词>"
```

**方法二：按修改的关键文件和函数搜索**
```bash
cd $DISTRO_GIT_REPO
# 搜索修改了关键文件的 commit
git log --all --oneline -- <被修复的源文件路径>
# 例如: git log --all --oneline -- arch/x86/kernel/alternative.c

# 搜索 commit message 中提到上游 commit hash 的
git log --all --oneline --grep="<上游 commit hash 前 12 位>"
```

**方法三：按关键代码变更搜索**
```bash
cd $DISTRO_GIT_REPO
# 搜索引入了修复函数/变量的 commit
git log --all -S "<修复引入的关键函数或变量名>" --oneline -- <相关文件>
# 例如: git log --all -S "int3_emulate_call" --oneline -- arch/x86/kernel/alternative.c
```

### 步骤 4.3.3：验证修复代码确实存在于新版本中

**找到疑似修复 commit 后，不能直接下结论。必须验证修复代码确实存在于发行版的某个已发布 tag 中。**

```bash
cd $DISTRO_GIT_REPO

# 1. 查看疑似修复 commit 的详细内容，确认确实是对应的修复
git show <疑似修复commit_hash>

# 2. 找出包含该 commit 的所有 tag（即哪些发行版版本包含了此修复）
git tag --contains <修复commit_hash> | sort -V

# 3. 确认当前 vmcore 的版本 tag 不在列表中（即确认当前版本未修复）
# 4. 确定列表中最早的 tag（即最早包含修复的发行版版本）
```

### 步骤 4.3.4：代码对比验证（关键！）

**进一步通过代码对比确认修复确实生效：**

```bash
cd $DISTRO_GIT_REPO

# 对比当前版本和修复版本中关键文件的差异
git diff <当前版本tag> <修复版本tag> -- <被修复的源文件路径>

# 或者直接查看修复版本中的代码，确认修复逻辑存在
git show <修复版本tag>:<被修复的源文件路径> | grep -A 10 "<修复的关键函数或代码>"
```

### 步骤 4.3.5：产出与报告

**产出格式：**

| 项目 | 内容 |
|------|------|
| 当前 vmcore 版本 | `<当前版本 tag>`（未修复） |
| 发行版修复 commit | `<commit hash>` — `<commit 标题>` |
| 对应的上游 commit | `<上游 commit hash>`（如适用） |
| 最早包含修复的 tag | `<最早修复 tag>` |
| 建议升级到 | `>= <最早修复 tag>` |
| 验证方式 | 通过 `git diff` / `git show` 确认修复代码存在 |

**在报告的"建议与修复"章节中，必须包含以下信息：**
- 发行版仓库中是否已有修复
- 如果已有：明确告知用户升级到哪个版本即可获得修复
- 如果没有：建议用户自行反向移植上游 commit 或向发行版维护者提交请求

### 常见场景处理

| 场景 | 处理方式 |
|------|---------|
| 发行版仓库中找到了修复 commit，且已包含在较新的 tag 中 | 报告修复版本，建议升级 |
| 发行版仓库中找到了修复 commit，但未包含在任何已发布 tag 中（仅在开发分支） | 说明修复已在开发中，建议关注下一个发行版版本 |
| 发行版仓库中**未找到**修复 commit | 说明发行版尚未反向移植此修复，建议用户自行反向移植上游 commit 或提交请求 |
| 上游修复 commit 本身尚未找到（步骤 4.1 无结果） | 跳过发行版检查，在报告中说明上游和发行版均未找到针对性修复 |
| `git tag --contains` 输出过多 tag（>50）| 使用 `sort -V | head -5` 只取最早的几个 tag，关注最早引入修复的版本 |
| 发行版使用了不同的修复方式（非直接 cherry-pick 上游 commit） | 通过代码对比确认修复逻辑等价即可，不要求 commit hash 完全一致 |
