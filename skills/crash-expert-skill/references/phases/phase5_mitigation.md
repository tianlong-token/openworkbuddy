# 阶段五：缓解措施分析（详细步骤）

从 **触发条件分析 → 业务侧规避建议 → 内核参数调整** 三个层面综合评估给出可行性建议。

---

## 🔴 第一步：触发条件分析（必须先做，再讨论缓解措施）

**在给出任何缓解建议之前，必须先基于阶段三的根因分析，梳理清楚 bug 的触发条件链——即哪些条件同时满足时才会触发此崩溃。**

### 分析框架

从以下维度逐一分析，哪些与本次崩溃的触发有因果关联：

| 维度 | 分析内容 | 示例 |
|------|---------|------|
| **应用行为/使用模式** | 应用进程的哪些操作或使用模式触发了问题路径？ | 高频 `madvise(MADV_HUGEPAGE)` 调用、大量短连接创建/销毁、频繁 `mmap`/`munmap`、高并发 `fork`等 |
| **内核特性/机制的使用** | 哪些内核特性被启用或被应用间接触发？ | |
| **并发/时序条件** | 是否需要特定的并发或时序才能触发？ | |
| **系统状态/资源条件** | 系统处于什么资源状态时才会触发？ | 内存压力（free 低于水位线）、内存碎片化等 |
| **配置条件** | 哪些系统/内核/应用配置是触发前提？ | |

### 产出要求

以**触发条件链**的形式写入报告，例如：

> **触发条件链：**
> 1. 系统启用了 ftrace 动态追踪（由监控 agent 的 kprobe 插桩触发）
> 2. 在 ftrace enable 过程中，`text_poke_bp()` 使用 int3 断点机制修改内核代码
> 3. 恰好另一个 CPU 执行到被修改的代码地址，触发 int3 异常
> 4. `poke_int3_handler` 在恢复执行地址时计算错误，跳转到非法地址
>
> **关键触发因素**：监控 agent 的 kprobe 动态插桩操作。如果停止该 agent 或减少插桩频率，可降低触发概率。

**🔴 核心要求：**
- 触发条件分析必须**具体到应用/业务层面**——不能只说"内核存在竞态"，要说清楚"什么样的业务操作/工具/使用模式导致这条代码路径被执行"
- 如果触发与特定应用进程/工具相关（如监控 agent、负载均衡器、容器运行时等），必须明确指出
- 如果触发需要多个条件同时满足，列出所有条件，帮助用户评估哪些条件最容易在业务侧规避

---

## 🔴 第二步：业务侧规避建议

## 🔴🔴🔴 最高优先级原则：不给无效建议，宁缺毋滥

**基于触发条件分析的结论，评估是否可以从业务侧规避 bug 的触发。**

| 规避方向 | 适用场景 | 示例 |
|---------|---------|------|
| **停用/替换触发组件** | 有明确的应用/工具触发了问题路径 | "暂停监控 agent 的 kprobe 插桩"、"将 iptables 规则迁移至 nftables" |
| **调整应用使用模式** | 应用的特定使用姿势触发了问题 | "减少并发 fork 频率"、"将 `madvise(MADV_HUGEPAGE)` 调用改为按需使用" |
| **降低触发概率** | 无法完全避免但可降频 | "降低 kprobe 插桩数量"、"错峰执行大内存分配操作" |
| **隔离故障影响** | 无法避免触发但可减少影响范围 | "为关键服务配置独立 cgroup"、"将故障进程限制在特定 CPU 集合" |

**🔴 注意：**
- 业务侧规避建议同样需要因果有效性——必须能从原理上解释"为什么该调整能避免或减少触发"
- 如果触发条件完全在内核侧（如纯内核定时器竞态，无业务操作参与），则不需要强行给出业务侧建议，但需在报告中说明"该问题的触发不依赖特定业务操作，无法从业务侧规避"

---

## 第三步：内核参数调整建议

## 🔴🔴🔴 最高优先级原则：不给无效建议，宁缺毋滥

**给出的每一条建议都必须经过严谨的因果分析，确认对解决或缓解当前问题有正向作用。如果从 vmcore 分析和原理分析后，确认不存在可以通过内核参数调整来解决的问题，则不需要给出任何内核参数调整建议。报告中"系统参数建议"一节可以为空或仅列出参数现状说明。**

**🔴 禁止"凑数式建议"**：不能因为报告需要"建议"章节就硬凑内核参数调整建议。没有有效的参数建议时，只给出业务侧建议即可。

## 🔴 强制前置步骤一：因果有效性验证（每条参数建议必做）

**在给出任何内核参数调整建议之前，必须先完成以下因果有效性自检。任何一项不通过，则该参数建议不得出现在报告中。**

对每一条计划建议的内核参数调整，必须回答以下 3 个问题：

| # | 自检问题 | 不通过则 |
|---|---------|---------|
| 1 | **因果关系**：修改此参数能否从原理上解决或缓解已定位到的根因？（需给出从"参数改变"到"问题缓解"的完整因果推导链） | 🚫 不给出此建议 |
| 2 | **副作用评估**：在当前系统配置下（物理内存大小、Swap 有无、业务实际内存需求等），修改此参数是否会引入新的问题或让情况更糟？ | 🚫 不给出此建议 |
| 3 | **现实可行性**：修改此参数后，系统是否仍然能正常运行业务？ | 🚫 不给出此建议 |

**典型反面案例（必须避免）：**

| 场景 | 错误建议 | 为什么错误 |
|------|---------|-----------|
| OOM（业务内存超配 118GB / 124GB 物理内存，无 Swap） | 建议设置 `vm.overcommit_memory=2` | 只是把"OOM 被杀"换成了"malloc 失败"，根本问题未解决 |

## ⚠️ 强制前置步骤二：配置验证

**在建议调整任何内核参数之前，必须先从 vmcore 中读取该参数的当前值，并在报告中列出"当前值 → 建议值"对比。**

具体要求：
1. **先查后建议**：对每个计划建议调整的参数，先用 crash 命令（如 `p <变量名>`）读取当前值
2. **当前值已等于建议值，则不输出该建议**，说明"当前配置已合理"
3. **无法读取某参数的值，必须说明"当前值未知"**，并提供查看方法（如 `cat /proc/sys/vm/xxx`）
4. **报告中以"当前值 vs 建议值"对比表呈现**

常见需要验证的参数示例：
- THP 配置：`p/x transparent_hugepage_flags`
- Compaction：`p sysctl_compaction_proactiveness`、`p sysctl_compact_unevictable_allowed`
- 内存水位：`p watermark_boost_factor`
- Watchdog：`p watchdog_thresh`、`p softlockup_panic`

## 🔴 内核模块/per-netns sysctl 参数查找方法（重要！）

**很多 sysctl 参数不是独立的全局变量，而是嵌入在 per-netns 或模块私有结构体中的字段。`sym` 命令和 `p` 命令无法直接搜索到这些参数。**

### ❌ 常见错误做法
```
crash> sym -q ignore_no_rs    → 找不到
crash> sym ignore_no_rs_error → symbol not found
→ 错误结论："该参数不存在"
```

### ✅ 正确查找流程

**第 1 步：查内核源码确认参数是否存在**
```bash
# 在内核源码中搜索参数名（procname 或字段名）
grep -rn "<参数关键词>" <内核源码目录>/include/net/
grep -rn "<参数关键词>" <内核源码目录>/net/
grep -rn "<参数关键词>" <内核源码目录>/<相关子系统>/
```

**第 2 步：确定参数所在的结构体和字段名**
- 找到 sysctl table 注册代码（`ctl_table` 数组），确认 `procname` 和 `data` 指向
- 找到字段定义所在的结构体（如 `netns_ipvs`、`net`、`netns_sysctl`）

**第 3 步：反汇编使用该参数的函数，获取偏移量**
```
crash> dis <使用该参数的函数>
# 从反汇编中找到 mov offset(%rax), %eax 指令，确认偏移量
```

**第 4 步：获取结构体实例指针，读取运行时值**
```
# 以 IPVS 参数为例：
crash> p init_net.ipvs           # 获取 netns_ipvs 指针
crash> p/d *(int *)(<指针> + <偏移量>)  # 读取参数值

# 以 net 命名空间参数为例：
crash> p init_net                # 获取 init_net 结构体
crash> struct net.ipv4 <init_net_addr>  # 获取子结构体
```

### 典型场景对照表

| sysctl 参数 | 存储位置 | 查找方式 |
|-------------|---------|---------|
| `vm.swappiness` 等全局参数 | 全局变量（`vm_swappiness`） | ✅ 可直接 `p vm_swappiness` |
| `kernel.watchdog_thresh` | 全局变量 | ✅ 可直接 `p watchdog_thresh` |
| `net.ipv4.vs.*`（IPVS 参数） | `struct netns_ipvs` 字段 | ❌ 不能用 sym，需源码+反汇编+偏移量 |
| `net.ipv4.conf.*.rp_filter` | per-netns `devinet_conf` | ❌ 不能用 sym，需 per-device 结构体 |
| `net.netfilter.*` | `struct netns_ct` 等 | ❌ 不能用 sym，需源码+偏移量 |
| 内核模块自定义 sysctl | 模块私有结构体 | ❌ 需先 `mod -s` 加载模块符号 |

## ⚠️ 区分"系统默认行为"与"进程主动请求"

当分析涉及 THP、madvise 等可由进程主动请求的特性时：
1. 先确认系统级配置（如 THP enabled=always/madvise/never）
2. 如果配置为 madvise 模式，说明只有进程通过 `madvise()` 主动请求才会触发，应将排查方向转向"哪个进程/库主动请求了该特性"
3. 报告中必须准确描述触发方式

## `hung_task_panic` 参数处理规则

- 当前值已为 1：无需额外提醒
- 当前值为 0：**禁止**建议开启 `kernel.hung_task_panic=1`
- 仅当用户主动询问时，说明风险：该参数会在 hung task 检测触发时直接 panic 导致系统重启

## 🔴 OOM/内存耗尽场景的建议禁区

**当 Panic 类型为 OOM 或根因涉及内存耗尽时，以下建议被严格禁止：**

- 🚫 **禁止建议开启或增加 Swap 空间**：Swap 不能根本性解决内存耗尽问题，只会将问题推迟并引入性能退化。内存耗尽的正确应对方式是**定位内存耗尽的根因**（哪个进程/模块消耗了过多内存、为什么消耗、如何限制），而非通过 Swap 缓冲来掩盖问题
- 🚫 **禁止建议调低 `vm.swappiness`**（在无 Swap 的系统上该参数无意义）

**OOM 场景的建议应聚焦于：**
1. **定位内存大户**：明确列出内存占用 top 进程，分析其内存使用是否合理
2. **分析内存耗尽原因**：是业务进程内存泄漏、配置不当（JVM 堆过大等）、还是内核态内存泄漏
3. **限制内存使用**：建议合理配置各组件内存上限、cgroup 内存限制。🔴 **禁止给出"内存上限之和不应超过物理内存的 X%"之类的具体比例建议**，因为合理的内存比例因业务场景而异，不具备普适性。只需列出当前各组件的内存占用事实，建议业务侧根据自身场景合理规划各组件的内存配额
4. **评估 OOM Killer 配置**：如存在 `oom_score_adj=-1000` 的进程，需指出其对 OOM Killer 选择的影响，建议业务侧评估该设置的合理性（注意：因为部分场景下这是业务需要，不可直接建议移除）
5. **防止被杀服务立即重启导致内存再次耗尽**：配置延迟重启策略
